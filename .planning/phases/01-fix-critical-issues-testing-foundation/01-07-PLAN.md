      2. Data Visibility (CR-02/FIX-02): filter logic not scoped to current user
      3. CORS Configuration (CR-03/FIX-03): hardcoded allowed origins
      4. Exception Handling (WR-01/FIX-04): unguarded exceptions in filter chain

      6. Error Silencing (WR-03/FIX-06): empty catch blocks

    - For each: pattern name, affected files, root cause, fix approach, prevention strategy
    - Include the 4 info items as lower-priority patterns: dead code (IN-01/IN-02), inconsistent error handling (IN-03), data class entities (IN-04)

    03-test-conventions.md:
    - Backend test patterns:
      - Framework: JUnit 5 + MockK (not Mockito)
      - Package mirror: src/test/kotlin/com/shareshelf/{module}/
      - File naming: {ClassName}Test.kt
      - Mock style: mockk<>() with every/verify, slot/capture
      - Constructor injection for service tests (no Spring context)
      - @WebMvcTest with @MockkBean for controller tests
      - assertThrows for error path testing
    - Frontend test patterns:
      - Framework: Vitest + React Testing Library
      - Config: vitest.config.ts with jsdom environment, @/ alias
      - Setup: src/test/setup.ts imports @testing-library/jest-dom
      - File naming: *.test.tsx (colocated in __tests__/ or alongside source)
      - Mock patterns: vi.mock for api, auth, next/navigation
      - Component test pattern: describe/it blocks, render/screen/fireEvent
      - Page test pattern: mock api.get/post/put, test all states (loading, error, empty, success, auth redirect)
    - Key conventions to follow in every future phase:
      - TDD-first: failing test before implementation
      - Unit tests only in Phase 1 (no @WebMvcTest or @DataJpaTest unless explicitly added)
      - Tests validate behavior, not implementation details (prefer behavior-driven assertions over mock verification)

    05-fix-approaches.md:
    - TDD cycle pattern used in Phase 1:
      1. Write failing test that describes expected behavior
      2. Verify test fails (RED)
      3. Write minimum code to pass
      4. Verify test passes (GREEN)
      5. Commit with conventional commit message
    - Common fix patterns:
      - Missing @Transactional: add annotation, verify with reflection test
      - Error silencing: replace empty catch with error state + user-facing message
      - Hardcoded data: replace with API fetch in useEffect
      - Render-time redirect: move to useEffect
      - !! assertions: replace with ?: throw IllegalStateException("descriptive message")
      - JWT filter exceptions: wrap in try-catch, clear context, always call filterChain.doFilter()
    - Preferred patterns to carry forward:
      - ApiResponse wrapper for all responses
      - GlobalExceptionHandler for error mapping
      - useEffect for side effects (not render phase)
      - Safe null handling with ?: throw
      - @Transactional on multi-write service methods
  </action>
