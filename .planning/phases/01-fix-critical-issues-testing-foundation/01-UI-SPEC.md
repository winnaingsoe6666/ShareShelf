## Typography

Phase 1 touches only 4 existing page files (`borrow/page.tsx`, `items/new/page.tsx`, `login/page.tsx`, `register/page.tsx`). The typography below is scoped to what these pages actually use. New pages in later phases may declare additional sizes.

Font sizes and weights are existing Tailwind v4 defaults consumed from production code; Phase 1 introduces NO new typography tokens.

| Role | Size | Weight | Line Height | Page Usage |
|------|------|--------|-------------|------------|
| Metadata | 12px (text-xs) | 400 (font-normal) | 1.25 (Tailwind text-xs) | `borrow/page.tsx` -- "Requested {date}" |
| Body | 14px (text-sm) | 400 (font-normal) | 1.25 (Tailwind text-sm) | All 4 pages -- form labels, descriptions, card metadata, paragraph text |
| Heading h2 | 20px (text-xl) | 600 (font-semibold) | 1.25 (Tailwind text-xl) | `login/page.tsx`, `register/page.tsx` -- page subtitle ("Welcome back", "Create your account") |
| Heading h1 | 24px (text-2xl) | 600 (font-semibold) | 1.2 (Tailwind text-2xl) | All 4 pages -- "My Borrows", "List a New Item", brand link, page title |

**Rationale for consolidation to 4 sizes and 2 weights:**

- **Sizes reduced from 5 to 4:** The 18px (text-lg) and 16px (text-base) and 36px (text-4xl) tokens exist in unmodified pages (homepage, items/browse) but are NOT consumed by any of the 4 Phase 1 affected pages. They are scoped out of this contract.
- **Weights reduced from 4 to 2:** `font-medium` (500) was used on labels and tab buttons -- remapped to `font-semibold` (600) to consolidate. `font-bold` (700) was used on page h1 headings -- also remapped to `font-semibold` (600) as the pixel size alone (24px) provides adequate visual hierarchy. The only weights Phase 1 requires are `font-normal` (400) for body/metadata and `font-semibold` (600) for headings, labels, and tabs.

**Font family:** System UI stack (Tailwind v4 default `ui-sans-serif, system-ui, ...`). No custom `@font-face` or Google Fonts configured.

The primary visual anchor on the borrow page is the **request card list**, with **tab navigation** acting as secondary navigation above it.

**Layout hierarchy (top to bottom):**
1. Navbar (global, persists across pages)
2. Page title: "My Borrows" (h1, 24px semibold, stone-900)

## Interaction Contracts

### Auth Redirect Guard (FIX-08)

**Pages affected:** `login/page.tsx`, `register/page.tsx`, `items/new/page.tsx`

**Current (BUG):** Auth check at render-time causes flash of protected content before redirect:
```tsx
// BAD -- executed during render, not effect
if (typeof window !== "undefined" && !isAuthenticated()) {
  router.push("/login");
  return null;
}
```

**Required (FIX):** Auth check in `useEffect` after mount:
```tsx
// GOOD -- executed after render in effect
useEffect(() => {
  if (typeof window !== "undefined" && !isAuthenticated()) {
    router.push("/login");
  }
}, [router]);
```

**Note on `items/new/page.tsx`:** This page has the render-time redirect bug AND uses `useEffect` for nothing currently. FIX-08 adds the redirect `useEffect` AND keeps the existing form render (there is no data fetching dependency -- the form renders fine unauthenticated because API calls will 401). The `borrow/page.tsx` already uses the correct `useEffect` pattern at lines 30-33 and does NOT need FIX-08 changes.

**Pattern applied in Phase 1:**
- `login/page.tsx`: Replace render-time guard with `useEffect` redirect. Redirect target: `/items` (if already logged in, send to browse page).
- `register/page.tsx`: Replace render-time guard with `useEffect` redirect. Redirect target: `/items` (if already logged in, send to browse page).
- `items/new/page.tsx`: Add `useEffect` with redirect to `/login` if not authenticated. Remove render-time guard.

**Redirect targets are NOT changed by Phase 1.** Only the mechanism changes (render time to useEffect).

### Borrow Page Tab Filter (FIX-02)

**Current (BUG):** Filter uses `r.borrowerId` and `r.ownerId` truthy checks which don't filter correctly:
```tsx
// BAD -- filters by existence of IDs, not current user's IDs
const filtered = tab === "borrowed"
  ? requests.filter((r) => r.borrowerId)
  : requests.filter((r) => r.ownerId);
```

**Required (FIX):** Filter against current authenticated user's ID:
```tsx
const user = getUser(); // from lib/auth
const filtered = tab === "borrowed"
  ? requests.filter((r) => r.borrowerId === user?.id)
  : requests.filter((r) => r.ownerId === user?.id);
```

**Visual effect:** No visual change to the UI. The filter logic change means each user sees only their own requests. The visual contract (tab headings, card layout, badge colors, action buttons) is unchanged.

### Borrow Page API Error Handling (FIX-06)

**Current (BUG):** Empty catch blocks swallow errors silently:
```tsx
// BAD -- error swallowed
api.get("/borrow")
  .then((res) => setRequests(res.data.data ?? []))
  .catch(() => {}) // silent failure
  .finally(() => setLoading(false));
```

**Required (FIX):** Error state tracked and rendered using the established error banner pattern:
```tsx
const [error, setError] = useState<string>("");

// In fetch
.catch((err) => {
  setError("Failed to load borrow requests. Please try again.");
})

// In render (between loading check and empty state)
{error && (
  <div className="rounded-lg bg-red-50 px-4 py-3 text-sm text-red-700">{error}</div>
)}
```

**Visual effect:** Error banner appears above the empty state when the API call fails. Currently the page would show a perpetual spinner (loading never set to false) or an empty list with no explanation. After fix, a red error banner explains the failure.

### Borrow Page Action Error Handling (FIX-06, handleAction)

**Current (BUG):** Empty catch in handleAction swallows errors:
```tsx
// BAD -- error swallowed
try {
  await api.put(`/api/borrow/${id}/${action}`);
  // optimistic update
} catch {
  // ignore
}
```

**Required (FIX):** Show error feedback without breaking optimistic UI:
```tsx
const [actionError, setActionError] = useState<string>("");

try {
  await api.put(`/api/borrow/${id}/${action}`);
  // optimistic update on success only -- revert on failure
} catch {
  setActionError(`Failed to ${action} request. Please try again.`);
  // Keep previous state (don't re-fetch entire list)
}
```

**Visual effect:** Error message appears at top of card list when action fails. After 5 seconds or on next tab switch, the error clears.

### Category Dropdown API-Driven (FIX-07)

**Current (BUG):** Hardcoded categories:
```tsx
<select>
  {["Tools", "Electronics", ...].map((name, i) => (
    <option key={i + 1} value={i + 1}>{name}</option>
  ))}
</select>
```

**Required (FIX):** Fetch categories from API in useEffect and render from state:
```tsx
const [categories, setCategories] = useState<Category[]>([]);
const [catLoading, setCatLoading] = useState(true);

useEffect(() => {
  api.get("/categories")
    .then((res) => setCategories(res.data.data ?? []))
    .catch(() => setError("Failed to load categories"))
    .finally(() => setCatLoading(false));
}, []);

// In the select
<select value={categoryId} onChange={...}>
  <option value="">Select a category</option>
  {catLoading ? (
    <option disabled>Loading categories...</option>
  ) : categories.length === 0 ? (
    <option disabled>No categories available</option>
  ) : (
    categories.map((cat) => (
      <option key={cat.id} value={cat.id}>{cat.name}</option>
    ))
  )}
</select>
```

**Visual effect:** Category dropdown now populates from API instead of hardcoded array. Visual appearance of the select element is unchanged (same Tailwind classes). Loading and error states added to the select options.

## Test Infrastructure Contract (TEST-02)

### Test Runner Configuration

| Property | Value |
|----------|-------|
| Framework | Vitest |
| DOM matchers | @testing-library/react + @testing-library/jest-dom |
| Config file | `frontend/vitest.config.ts` |

| Test file pattern | `*.test.tsx` (colocated with source or in `__tests__/`) |
| Mock strategy | vi.mock for external deps (api, auth, router) |

### Component Test Requirements

Each UI component test MUST:
1. Import and render the component
2. Verify default state renders without error
3. Verify each variant/state produces correct CSS classes
4. Verify event handlers fire correctly (onClick, onChange)
5. Verify conditional rendering (loading, error, disabled)

### Page Test Requirements

Each page test MUST:
1. Mock `api` and `auth` dependencies with `vi.mock`
2. Test render success state (mocked API returns data)
3. Test loading state (skeleton/spinner visible)
4. Test error state (error banner visible with correct message)
5. Test empty state (empty state copy visible)
6. Test auth redirect (unauthenticated user redirected)
7. Test tab filtering (borrow page -- FIX-02 verification)

### Mock Patterns

```tsx
// Mock API client
vi.mock("@/lib/api", () => ({

// Mock Next.js router

  useRouter: vi.fn(() => ({
    push: vi.fn(),

  })),
```
