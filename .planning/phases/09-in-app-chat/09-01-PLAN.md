<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → WebSocket | STOMP CONNECT carries JWT — must validate before allowing subscribe/send |
| STOMP message → Server | Message payload crosses WebSocket channel — must validate sender identity matches authenticated user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Spoofing | WebSocketAuthInterceptor | mitigate | Validate JWT on STOMP CONNECT, reject unauthenticated connections |
| T-09-02 | Tampering | STOMP message payload | mitigate | Server sets senderId from authenticated principal, not from client payload |
| T-09-03 | Information Disclosure | Chat messages | mitigate | Users can only subscribe to their own /topic/chat/{userId}, enforce participant check in repository queries |
| T-09-04 | Denial of Service | Message send endpoint | accept | Rate limiting deferred per CONTEXT.md; add if abuse occurs |
| T-09-05 | Elevation of Privilege | Chat access | mitigate | Verify user has made a borrow request for the item before allowing message send (enforced in service layer, next plan) |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | @stomp/stompjs and sockjs-client are well-known packages; verify legitimacy before install in frontend plan |
</threat_model>
