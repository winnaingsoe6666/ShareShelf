<objective>
Build the chat service layer, REST endpoints, and STOMP message controller for real-time messaging.

Purpose: Provides the backend API that the frontend (plan 09-03) consumes. REST endpoints serve conversation history and inbox data. STOMP controller handles real-time message delivery — persisting messages to PostgreSQL first, then broadcasting to the recipient's STOMP topic. Chat access is gated by borrow request existence per D-01.

Output:
- ChatDtos.kt with SendMessageRequest, MessageResponse, ConversationResponse, UnreadCountResponse
- ChatService with conversation retrieval, message sending, mark-as-read, and unread count logic
- ChatController with REST endpoints: GET /conversations, GET /conversations/{itemId}/{otherUserId}, POST /conversations/{itemId}/{otherUserId}/read, GET /unread-count
- ChatStompController with @MessageMapping("/chat.send") that persists + broadcasts via SimpMessagingTemplate
</objective>

    **File 2: ChatStompController.kt** (package: com.shareshelf.chat)

    @Controller class (NOT @RestController — STOMP uses @Controller). Constructor injects ChatService and SimpMessagingTemplate (org.springframework.messaging.simp.SimpMessagingTemplate).

    Methods:

    - `@MessageMapping("/chat.send")` fun sendMessage(@Payload request: SendMessageRequest, principal: java.security.Principal): Unit`
      - Extract userId from principal (Spring Security sets this from WebSocketAuthInterceptor in plan 09-01)
      - The principal.getName() returns the userId string (set by WebSocketAuthInterceptor)
      - Calls chatService.sendMessage(userId.toLong(), request) to persist to DB
      - Broadcasts the saved MessageResponse to `/topic/chat/${request.receiverId}` via simpMessagingTemplate.convertAndSend()
      - This ensures offline users get the message on next REST history load (persisted first), and online users get it instantly via STOMP

    Important: The STOMP controller does NOT return a value (void). It sends the message to the topic asynchronously. The sender also receives confirmation because they subscribe to their own topic `/topic/chat/{senderId}` (handled in frontend plan 09-03).

    Also broadcast to sender's topic so sender gets the message back (for multi-tab sync):
    - `simpMessagingTemplate.convertAndSend("/topic/chat/${saved.senderId}", messageResponse)`
    - `simpMessagingTemplate.convertAndSend("/topic/chat/${saved.receiverId}", messageResponse)`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → REST API | JWT Bearer token in Authorization header — Spring Security validates |
| Browser → WebSocket | JWT in STOMP CONNECT header — WebSocketAuthInterceptor validates (plan 09-01) |
| STOMP message → Server | Message payload crosses WebSocket channel — server must set senderId from authenticated principal, not from client payload |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-06 | Tampering | ChatStompController | mitigate | Sender ID extracted from java.security.Principal (set by WebSocketAuthInterceptor), NOT from the SendMessageRequest payload. Client cannot spoof sender. |
| T-09-07 | Elevation of Privilege | ChatService.sendMessage | mitigate | Verify sender has a borrow request for the item OR is the item owner before allowing message send. AccessDeniedException if neither. |
| T-09-08 | Information Disclosure | ChatController endpoints | mitigate | All endpoints require @AuthenticationPrincipal. getConversation verifies user is a participant. Users cannot read others' conversations. |
| T-09-09 | Denial of Service | Message send | accept | Rate limiting deferred per CONTEXT.md. Message length capped at 2000 chars as basic mitigation. |
</threat_model>
