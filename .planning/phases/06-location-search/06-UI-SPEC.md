## New Components

### 1. LocationPicker

**Purpose:** Map widget embedded in item create/edit forms. User clicks or drags to place a single marker representing the item's location.

**File:** `frontend/src/components/map/LocationPicker.tsx`

**Props:**
```typescript
interface LocationPickerProps {
  latitude?: number | null;   // pre-existing lat (edit mode)
  longitude?: number | null;  // pre-existing lng (edit mode)
  onChange: (lat: number, lng: number) => void;  // called when pin position changes
  onClear: () => void;        // called when pin is removed
  error?: string;             // external error message
}
```

**States:**

| State | Visual Output | Test Verifies |
|-------|---------------|---------------|
| Default (no pin) | Map centered on default location. Skeleton map container while tiles load. "Click on the map to set the location" overlay centered on map area. No marker visible. | Overlay text rendered, onChange not called, map container present |
| Default center | Map centered at 16.84, 96.17 (Yangon). Zoom level 13. | MapContainer receives center={[16.84, 96.17]} and zoom={13} |
| Geolocation available | Browser requests permission on mount via `navigator.geolocation.getCurrentPosition`. On success, map recenters to user's position. On deny, stays at default center. | Map center updates to user coords when granted |
| Pin placed (click) | Marker appears at clicked position. `onChange(lat, lng)` fires. Overlay text disappears. "Remove pin" link appears below map. | Marker rendered at click coordinates, onChange called with correct coords |
| Pin placed (edit mode) | Map loads with marker at `latitude, longitude`. Overlay hidden. "Remove pin" link visible. | Marker rendered at provided coords, overlay absent |
| Pin moved (drag) | Marker is draggable. On dragend, `onChange(newLat, newLng)` fires with updated position. | Marker has draggable={true}, dragend triggers onChange |
| Pin removed | Marker removed. Overlay text reappears. `onClear()` called. "Remove pin" link disappears. | Marker not in DOM, onClear called |
| Map load error | Non-blocking gray container with "Map unavailable" message. Form still functional. | Error message rendered, form inputs still present |
| Tight space (>320px wide) | Map height reduces to 200px. Overlay text and controls adjust spacing. | Container has min-h-[200px], responsive classes |

**Layout:** Map container uses `rounded-xl border border-purple-200 overflow-hidden` to match Card styling. Height: 300px (desktop), 200px (mobile). "Remove pin" link styled as `text-sm text-purple-600 hover:text-purple-800 cursor-pointer inline-flex items-center gap-1` with `MapPinOff` lucide icon.

**Map tile style:** CartoDB Positron (muted, no-labels variant: `https://{s}.basemaps.cartocdn.com/light_nolabels/{z}/{x}/{y}{r}.png`). Matches the purple/soft design aesthetic without competing map colors.

### 6. DistanceBadge (on ItemCard)

**Purpose:** Shows distance when location filter is active. Appended to existing ItemCard layout.

**Location:** Between the category name line and the price/owner row on ItemCard.

**Style:**
```html
<div class="mt-2 inline-flex items-center gap-1 text-xs text-stone-500">
  <MapPin className="h-3 w-3" />
  <span>1.2 km away</span>
</div>
```

**Visibility:** Only rendered when `item.distance` is defined (i.e., when spatial query params were used). Absent from the DOM otherwise.

**Distance formatting rules:**
- `< 0.1 km` → "Nearby" (same neighborhood)

- `< 0.01 km` → "Nearby" (essentially same location)

**Reverse-geocode label:** When Nominatim reverse lookup succeeds, use it as a secondary label: "Near {suburb/neighborhood}" below the distance. When it fails or is unavailable, show only the distance. Nominatim is called with a 1-second rate limit (free tier requirement, 1 req/sec).

## Interaction Contracts

### Browse Page: Distance Filter Integration

**Location:** In the existing filter bar at `frontend/src/app/[locale]/items/page.tsx`, between the status filter group and the `|` separator before the rating filter.

**State variables added:**
```typescript
const [userLat, setUserLat] = useState<number | null>(null);
const [userLng, setUserLng] = useState<number | null>(null);
const [radius, setRadius] = useState<number | null>(null);  // null = no distance filter
const [geoLoading, setGeoLoading] = useState(false);
const [geoDenied, setGeoDenied] = useState(false);
```

**API params added (when userLat, userLng, radius all non-null):**
```typescript
// In the params object passed to api.get("/items", { params })
nearLat: userLat,
nearLng: userLng,
nearRadius: radius,
```

**Flow:**
1. User clicks "Use My Location" → `setGeoLoading(true)` → `navigator.geolocation.getCurrentPosition()`
2. Success → `setUserLat/lng` → `setRadius(5000)` (default 5km) → `setGeoLoading(false)`
3. Error/denied → `setGeoDenied(true)` → `setGeoLoading(false)` — radius presets still clickable (user pans map manually later)
4. User clicks radius preset → `setRadius(value)` — API re-fetches
5. User clicks "Any distance" → `setRadius(null)` — spatial filter removed, all items shown without distance

### Map Page: Data Fetching

**Strategy:** Bounds-based fetching with user location awareness.
- On mount: request geolocation. If granted, fetch items within 5km of user location. If denied, fetch items within default map bounds (Yangon center, zoom 13).
- On bounds change (pan/zoom, debounced 500ms): fetch items within current bounds + active radius.
- The API is called with `nearLat`, `nearLng`, `nearRadius` when radius is set, or with bounds-based params when radius is null.
- Search text filter applied alongside spatial params (combined query).

**API endpoint:** `GET /api/items` with added params `nearLat`, `nearLng`, `nearRadius`.

### Item Create/Edit: LocationPicker Integration

**Location:** In `frontend/src/app/[locale]/items/new/page.tsx` and the future edit page.

**Form integration:**
- `latitude` and `longitude` fields added to form state (both `number | null`, default `null`)
- `LocationPicker` component inserted between the price fields and the submit button
- On `onChange(lat, lng)`: update form state with both values
- On `onClear()`: set both to `null`
- On form submit: include `latitude` and `longitude` in the POST/PUT body (omitted if null)

**Existing form context preserved:**
- ImageUpload already uses a similar "add a widget to the form" pattern
- The LocationPicker follows that same pattern: a labeled section with a bordered component that handles its own internal state

### Geocoding: Reverse-Geocode Display Strategy

**Provider:** Nominatim (OpenStreetMap, free, no API key).
**Rate limit:** 1 request per second (client-side throttle).
**Cache:** In-memory Map keyed by rounded lat/lng (3 decimal places, ~111m precision). Cache persists for page session.
**Fallback:** On rate limit or failure, display only the distance ("X.X km away") without the neighborhood label.
**Display format:** "Near {suburb}, {city}" or "Near {neighbourhood}" depending on Nominatim response richness.
