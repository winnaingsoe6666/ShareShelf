<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → ItemController.listItems | nearLat/nearLng/nearRadius are user-controlled query parameters — untrusted input |
| ItemService → PostGIS | ST_DWithin and ST_MakePoint receive parameters from user input — SQL injection risk mitigated by parameterized query |
| HTTP client → ItemController.createItem | latitude/longitude in request body cross trust boundary — validated at DTO level by Bean Validation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Denial of Service | ItemController nearRadius | mitigate | Server-side validation rejects nearRadius > 50000m (50km). GiST index keeps legitimate queries fast, but bomb-radius query returning the entire table is prevented by the max radius cap. |
| T-06-05 | Tampering | nearLat/nearLng query params | mitigate | Invalid coordinate values (NaN, Infinity, extreme values) caught by `Double?` type coercion. Non-double strings are rejected by Spring's type conversion with 400 response. PostGIS `ST_MakePoint` with parameterized query prevents SQL injection. |
| T-06-06 | Information Disclosure | Error messages | accept | `IllegalArgumentException` messages ("nearLat and nearLng are required...") are returned to the client — acceptable for a public filter parameter. No sensitive data leaked. |
| T-06-07 | Tampering | ItemService.create() Point construction | mitigate | JTS Coordinate with valid Double values from Bean-validated CreateItemRequest. PostGIS geometry type rejects invalid SRID 4326 geometries at INSERT time. |
</threat_model>
